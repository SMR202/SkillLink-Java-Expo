B.3  Structured Specifications 
The following four structured specifications cover key user stories developed in Sprint 2. Each spec 
details the precise inputs, processing logic, outputs, pre/postconditions, and exception handling. 
 
Spec-05.1 — Full-Text Provider Search 
Spec ID 
Spec-05.1 
Description 
Accept a user-supplied query string and return a ranked list of matching 
service providers. 
Preconditions 
User is authenticated as a Client. Search index is available and synced with 
the provider database. 
Inputs 
query: string (1–100 chars). Optional filters: categoryId (integer), city (string), 
minRating (float 1.0–5.0), page (integer, default 1). 
Processing 
1. Sanitise and trim the query string (strip special chars). 2. Execute full-text 
search across provider name and skill tag fields. 3. Apply category, city, and 
rating filters as WHERE clauses. 4. Rank results by rating DESC, then 
completeness of profile. 5. Paginate: return 10 results per page with total 
count. 
Outputs 
JSON array of provider objects: { providerId, name, avatar_url, topSkill, 
avgRating, city }. Pagination metadata: { page, totalPages, totalResults }. 
Exceptions 
EmptyQuery: return validation error 'Please enter a search term'. NoResults: 
return empty array with message 'No providers found — try broadening your 
search'. DBTimeout: return HTTP 503 with retry message. 
 
Spec-06.1 — Public Provider Profile Rendering 
Spec ID 
Spec-06.1 
Description 
Fetch and render a complete provider profile page for a given provider ID. 
Preconditions 
Provider account exists and is active. Client is authenticated. 
Inputs 
providerId: integer (path parameter). viewerId: integer (from JWT session). 
Processing 
1. Validate providerId exists in DB; return 404 if not found. 2. Fetch provider 
record including bio, skills, portfolio URLs, and profile image URL. 3. Fetch 
aggregate rating: AVG(rating) from reviews table WHERE providerId. 4. Fetch 
latest 10 reviews ordered by createdAt DESC. 5. Log a profile view event for 
analytics. 6. Return assembled profile DTO to frontend. 
Outputs 
Provider profile page rendered with: avatar, name, bio, skill badges, portfolio 
icons, star rating (X.X / 5.0), review count, individual reviews, and 'Book Now' 
button. 
Exceptions 
ProviderNotFound (404): display 'This provider does not exist.' 
ProviderSuspended: show 'Account currently under review.' No reviews: 
display 'No reviews yet — be the first to book!'

Spec ID 
Spec-07.1 
Description 
Validate and persist a new booking request submitted by a Client. 
Preconditions 
Client is authenticated. Provider profile is active. Client has completed their 
contact profile (US-04). 
Inputs 
providerId: integer. clientId: integer (from JWT). preferredDate: date string 
(YYYY-MM-DD). preferredTime: time string (HH:MM). jobDescription: string 
(20–1000 chars). 
Processing 
1. Validate preferredDate is not in the past; reject if so. 2. Validate 
jobDescription length (>= 20 chars). 3. Check provider is not already booked 
at that slot (conflict check). 4. Insert booking record: { clientId, providerId, 
preferredDate, preferredTime, jobDescription, status: 'Pending', createdAt: 
now() }. 5. Dispatch in-app notification to provider. 6. Return booking 
confirmation with bookingId. 
Outputs 
HTTP 201 with booking object: { bookingId, status: 'Pending', providerId, 
preferredDate, preferredTime }. In-app notification delivered to provider. 
Exceptions 
PastDate: 'Selected date has already passed.' DescriptionTooShort: 'Please 
provide at least 20 characters describing your job.' SlotConflict: 'Provider is 
unavailable at this time — please choose another slot.' ProviderInactive: 'This 
provider is currently unavailable.' 
 
Spec-08.1 — Booking Accept / Decline by Provider 
Spec ID 
Spec-08.1 
Description 
Allow a Service Provider to accept or decline a pending booking request. 
Preconditions 
Provider is authenticated. Booking exists with status 'Pending'. Booking 
belongs to this provider. 
Inputs 
bookingId: integer. action: enum { 'accept', 'decline' }. declineReason: string 
(optional, max 200 chars, required only when action = 'decline'). 
Processing 
1. Fetch booking by bookingId; verify it belongs to the authenticated provider. 
2. Verify current status is 'Pending'; reject if already actioned. 3. If action = 
'accept': update status to 'Accepted', send confirmation notification to client. 4. 
If action = 'decline': update status to 'Declined', attach declineReason, send 
decline notification to client. 5. Log action timestamp. 
Outputs 
HTTP 200 with updated booking object: { bookingId, status: 'Accepted' | 
'Declined', updatedAt }. Push notification delivered to client. 
Exceptions 
BookingNotFound (404): 'Booking does not exist.' UnauthorisedProvider: 'You 
are not authorised to action this booking.' AlreadyActioned: 'This booking has 
already been accepted or declined.' DeclineReasonMissing: 'Please provide a 
reason when declining a booking.'

C.1  Introduction 
SkillLink is a digital service marketplace platform connecting clients who need skilled professionals 
with verified service providers across trades and expertise areas in Pakistan. The platform 
addresses a gap in the local market: the absence of a trustworthy, centralised portal where clients 
can discover, vet, and seamlessly book providers such as plumbers, electricians, graphic 
designers, tutors, and software developers. 
 
Sprint 1 established the foundational architecture for user onboarding and role-based profile 
management, delivering three of four planned user stories (US-01: Account Registration, US-02: 
Secure Authentication, US-03: Provider Profile Setup). US-04 (Client Contact Management) was 
carried forward as a leftover. 
 
Sprint 2 builds directly on this foundation. It delivers the leftover US-04 alongside four new user 
stories (US-05 through US-08), introducing the Search & Discovery and Booking & Scheduling 
modules. Together, these modules form the core transactional layer of the SkillLink platform — 
enabling clients to find providers and initiate service engagements. 
 
C.2  Project Vision 
The project vision established in Deliverable 1 remains unchanged and continues to guide all 
development decisions: 
 
To become the leading digital service marketplace in Pakistan, empowering clients 
to access quality services instantly while enabling skilled providers to grow their 
businesses through a transparent, secure, and user-friendly platform. 
 
Sprint 2 directly advances this vision by making the platform's search, discovery, and booking 
capabilities operational — transforming SkillLink from a registration system into a functioning 
marketplace.
